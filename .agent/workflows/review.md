# Review

Review content and deliverables against brand standards, brief requirements, and quality bar. Use this to check work before publishing or finalising.

## Step 1: Scope

Brainstorm with the user to define what we're reviewing:

```
What are we reviewing today?

1. A specific deliverable — [name or describe it]
2. A batch of recent work — everything since [date/session]
3. A full workspace review — check everything in output/
4. Something else — [describe]
```

**Wait for user response.**

Follow up to clarify:
- Which files or folders specifically?
- What standard are we reviewing against? (brand voice, accuracy, completeness, etc.)
- Any specific concerns to focus on?

## Step 2: Gather Context

Before reviewing, understand the standard:

| Source | What to look for |
|--------|-----------------|
| `brand/` | Brand voice, persona, visual identity — the baseline for "on brand" |
| Track brief (if exists) | What was the original brief? What was promised? |
| `conductor/strategy.md` | Overall goals and audience — does this serve the strategy? |

If `brand/` doesn't exist yet, note this — you'll be reviewing against general quality rather than brand-specific standards.

## Step 3: Review Criteria

Apply criteria based on what's being reviewed:

### Content (articles, scripts, posts, captions)
- [ ] **Brand voice** — does it sound like us?
- [ ] **Accuracy** — are facts, names, stats correct?
- [ ] **Engagement** — does it hook, hold, and reward attention?
- [ ] **Format** — proper structure (hook → body → CTA where applicable)
- [ ] **Length** — appropriate for the format and platform

### Strategy Documents
- [ ] **Completeness** — does it cover what it needs to?
- [ ] **Actionability** — can someone act on this, or is it vague?
- [ ] **Alignment** — does it serve the stated vision and goals?
- [ ] **Clarity** — is it easy to understand and reference later?

### Research & Analysis
- [ ] **Source quality** — are claims supported?
- [ ] **Coverage** — does it address the question fully?
- [ ] **Conclusions** — are they supported by the evidence presented?
- [ ] **Usefulness** — can this inform decisions?

### General (any deliverable)
- [ ] **Clarity** — is it well-organised and easy to follow?
- [ ] **Completeness** — is anything missing or half-done?
- [ ] **Appropriate length** — not too long, not too short for purpose
- [ ] **File location** — is it saved where it belongs?

## Step 4: Execute Review

Go through each piece systematically. For each deliverable:

1. Read it fully
2. Note strengths — what's working well
3. Note issues — what needs attention, with specifics
4. Rate: ✅ Ready / 🟡 Needs minor work / 🔴 Needs significant revision

**Important:** During this phase, observe and record. Don't fix things yet — complete the full review first.

## Step 5: Report

Present findings in a structured format:

```
# Review Report

## Summary
- Reviewed: [N items]
- Ready: [X] | Needs work: [Y] | Needs revision: [Z]

## Item-by-Item

### [Deliverable Name]
**Rating:** ✅ / 🟡 / 🔴

**Strengths:**
- [What's working]

**Issues:**
- [What needs attention — be specific]

**Suggestions:**
- [Concrete improvement recommendations]
```

**Wait for user response.**

## Step 6: Act

Based on user feedback:

```
How would you like to handle the findings?

1. Fix now — I'll make the changes to flagged items
2. Add to roadmap — note these for a future session
3. Note and move on — log the findings in memory
4. Some of each — [specify which items to fix vs. defer]
```

For items being fixed now:
1. Make the change
2. Re-read to verify the fix doesn't introduce new issues
3. Mark as resolved in the report

For items being deferred:
1. Log in memory.md or add to roadmap as appropriate
